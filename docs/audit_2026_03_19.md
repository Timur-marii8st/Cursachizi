## ЧАСТЬ 1: SECURITY — КРИТИЧЕСКИЕ УЯЗВИМОСТИ

### [SEC-C1] Незащищённый GET /api/jobs/{id} — утечка данных пользователей
**Файл:** `backend/app/api/routes/jobs.py:151-160`
**Серьёзность:** CRITICAL
Эндпоинт не требует аутентификации. Любой может получить полные данные любого задания (тема, университет, статус, S3-ключ документа) по UUID.

### [SEC-C2] Незащищённый GET /api/jobs/{id}/download — кража документов
**Файл:** `backend/app/api/routes/jobs.py:216-255`
**Серьёзность:** CRITICAL
Документы `.docx` доступны без авторизации. Перебор UUID позволяет скачать все сгенерированные работы платформы.

### [SEC-C3] Незащищённый POST /api/jobs/{id}/reference — порча шаблонов
**Файл:** `backend/app/api/routes/jobs.py:258-310`
**Серьёзность:** CRITICAL
Загрузка reference-шаблона не проверяет владельца задания. Любой может подменить шаблон чужого задания или исчерпать хранилище.

### [SEC-C4] Missing `await db.commit()` в cancel_job — потеря кредитов
**Файл:** `backend/app/api/routes/jobs.py:183-213`
**Серьёзность:** CRITICAL
Эндпоинт вызывает `flush()` но не `commit()`. При разрыве соединения после flush задание помечается CANCELLED, но кредиты не возвращаются → безвозвратная потеря средств пользователя.

### [SEC-C5] Missing `await db.commit()` в robokassa_result — потеря платежей
**Файл:** `backend/app/api/routes/payments.py:100-178`
**Серьёзность:** CRITICAL
Финансовая транзакция: кредиты зачисляются в `flush()` → Robokassa получает "OK" → при сбое соединения деньги списаны, кредиты не зачислены.

### [SEC-H1] list_jobs возвращает задания ВСЕХ пользователей
**Файл:** `backend/app/api/routes/jobs.py:163-180`
**Серьёзность:** HIGH
`GET /api/jobs` без `telegram_id` возвращает все задания платформы постранично.

### [SEC-H2] Rate limit bypass через X-API-Key ротацию
**Файл:** `backend/app/api/deps.py:63-116`
**Серьёзность:** HIGH
Ограничение скорости привязано к `X-API-Key`, который не валидируется перед использованием как client_id. Ротация произвольных строк в заголовке обходит rate limit полностью.

### [SEC-H3] Timing attack на INTERNAL_API_KEY
**Файл:** `backend/app/api/deps.py:127`
**Серьёзность:** HIGH
`if x_api_key != expected_key` — простое сравнение строк вместо `hmac.compare_digest()`.

### [SEC-H4] Health endpoint раскрывает детали инфраструктуры
**Файл:** `backend/app/api/routes/health.py:22-64`
**Серьёзность:** HIGH
`/api/health/ready` возвращает полные сообщения об ошибках подключения к БД и Redis в ответе.

## ЧАСТЬ 2: ASYNC/DB/WORKERS — КРИТИЧЕСКИЕ БАГИ

### [ASYNC-C1] TCP-утечка: AnthropicProvider и OpenAIProvider не реализуют aclose()
**Файлы:** `backend/app/llm/anthropic.py`, `backend/app/llm/openai_provider.py`
**Серьёзность:** CRITICAL
`tasks.py` вызывает `aclose()` в finally-блоке для всех провайдеров, но эти два метод не реализуют. Каждое задание утекает 2-5 TCP-соединений. После 100 заданий → исчерпание file descriptors → сервис недоступен.

### [ASYNC-C2] LLM клиенты создаются без async context manager протокола
**Файлы:** `backend/app/llm/anthropic.py:16`, `backend/app/llm/openai_provider.py:16`
**Серьёзность:** CRITICAL
SDK клиенты поддерживают `async with`, но не используются как context managers → некорректная очистка ресурсов.

### [ASYNC-H1] get_vision_llm_provider() создаёт новый экземпляр при каждом вызове
**Файл:** `backend/app/api/deps.py:174-184`
**Серьёзность:** HIGH
Каждый вызов создаёт новый `OpenRouterProvider` с новым httpx client pool без закрытия предыдущего.

### [ASYNC-H2] FSM состояние бота очищается ДО успеха API вызова
**Файл:** `bot/app/handlers/generate.py:163-201`
**Серьёзность:** HIGH
`await state.clear()` на строке 171 вызывается до `await api_client.create_job()`. При ошибке API пользователь теряет введённые данные и вынужден начинать заново.

### [ASYNC-M1] CourseForgeAPIClient не закрывается при остановке бота
**Файл:** `bot/app/main.py:39-63`
**Серьёзность:** MEDIUM
Нет shutdown hook для `await api_client.aclose()` → утечка httpx connection pool при каждом перезапуске.

### [ASYNC-M2] JobProgressCallback создаёт новую сессию на каждое обновление
**Файл:** `backend/app/workers/tasks.py:57-74`
**Серьёзность:** MEDIUM
6 стадий × 10+ обновлений = 60+ сессий за задание. При пиковой нагрузке пул соединений истощается.

## ЧАСТЬ 3: PIPELINE — ЛОГИЧЕСКИЕ БАГИ

### [PIPE-C1] Модуло-фолбэк в citation_fixer создаёт схождение цитат (many-to-one)
**Файл:** `backend/app/pipeline/writer/citation_fixer.py:368`
**Серьёзность:** CRITICAL
`(cited_num % registry_size) + 1` — при registry_size=5 цитаты [6], [11], [16] все → [1]. Разные источники тихо маппятся на один → порча библиографии без предупреждения.

### [PIPE-C2] Немаппированные цитаты тихо теряются без логирования
**Файл:** `backend/app/pipeline/writer/citation_fixer.py:298-302`
**Серьёзность:** HIGH
`_find_best_match()` возвращает `None` при score < 0.15 — цитата пропускается, запись в mapping не создаётся, никакого warning в логах.

### [PIPE-C3] format_with_content() не находит источники после URL-нормализации
**Файл:** `shared/schemas/pipeline.py:149-164`
**Серьёзность:** MEDIUM
Ranker нормализует URL (`https://example.com` → `example.com`), но BibliographyRegistry хранит оригинальный URL → lookup по `by_url` всегда промахивается.

### [PIPE-H1] _split_reference_block() вызывается дважды, результат первого вызова выброшен
**Файл:** `backend/app/pipeline/formatter/reference_extractor.py:128-132`
**Серьёзность:** HIGH
Copy-paste ошибка: двойной вызов, `body_text` переприсваивается. Производительность и читаемость кода страдают.

### [PIPE-M1] Некоторые секции могут получить меньше min_per_section источников
**Файл:** `backend/app/pipeline/writer/stage.py:194-204`
**Серьёзность:** MEDIUM
While-цикл топапа источников корректен по логике, но при исчерпании `all_nums` выходит без гарантии достижения минимума.

## ЧАСТЬ 4: PERFORMANCE

### [PERF-C1] Секции пишутся ПОСЛЕДОВАТЕЛЬНО — нет параллелизма
**Файл:** `backend/app/pipeline/writer/stage.py:123-146`
**Серьёзность:** CRITICAL
10-15 секций × 10-15 сек каждая = 100-225 сек. С параллелизмом (semaphore=3): 30-75 сек. **-50% latency для самой длинной стадии.**

### [PERF-H1] format_with_content() вызывается заново для КАЖДОЙ секции
**Файл:** `shared/schemas/pipeline.py:137-175` + `section_writer.py:200`
**Серьёзность:** HIGH
15 секций × полный проход по всем 20-30 источникам с построением dict = избыточная сериализация. Кешировать результат на уровне Registry.

### [PERF-H2] Поиск цитат последователен при fact-checking
**Файл:** `backend/app/pipeline/verifier/stage.py:71-88`
**Серьёзность:** HIGH
100 claims × 5 сек = 500 сек. С batching (semaphore=5): 100 сек.

### [PERF-H3] len(content.split()) вызывается 50+ раз за пайплайн
**Серьёзность:** HIGH
Распределён по 15+ файлам. Кешировать в `SectionContent.word_count` сразу при создании.

### [PERF-M1] Поисковые запросы выполняются последовательно
**Файл:** `backend/app/pipeline/research/stage.py:68-73`
**Серьёзность:** MEDIUM
10 запросов × rate_limit=1.0 сек = ~10 сек ожидания. Батчинг 3-5 параллельных запросов даст -60% latency.

|--------|----------------|

### Ключевые пробелы в edge-case тестировании

- Пустой ответ LLM → что происходит в pipeline?
- Пустые результаты поиска → 0 источников для раздела
- Concurrent job creation одного пользователя (race condition в user_service)
- Идемпотентность Robokassa webhook (дублирующий вызов)
- Таймаут воркера на каждой из 6 стадий
- Атомарность кредитного списания под concurrent load

## ЧАСТЬ 6: АРХИТЕКТУРНЫЕ РИСКИ

### [ARCH-1] Redis — единая точка отказа (SPOF)
Rate limiting, arq queue, bot sessions — всё на Redis. Падение Redis = каскадный отказ сервиса.

### [ARCH-2] Worker throughput ограничен 3 конкурентными заданиями
При росте нагрузки очередь растёт линейно. Нет приоритизации.

### [ARCH-3] Нет observability — метрики, трейсинг, алёрты
Нет отслеживания: стоимость токенов на задание, latency по стадиям, success rate, загрузка воркера.

### [ARCH-4] Нет retry на уровне воркера
Транзиентные ошибки (rate limit LLM, сетевой таймаут) → задание сразу FAILED → потеря кредита пользователя.
